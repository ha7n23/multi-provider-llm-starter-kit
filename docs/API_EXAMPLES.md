# API Examples

This document shows example requests and responses for the Multi-Provider LLM Starter Kit API.

The API is built with FastAPI and can be tested through the interactive Swagger UI:

```text
http://127.0.0.1:8000/docs
```

## Start the API Locally

Run with Python:

```bash
PYTHONPATH=src uvicorn multi_provider_llm.api.app:app --reload
```

Run with Docker:

```bash
docker run --rm -p 8000:8000 multi-provider-llm-starter-kit
```

Run Docker with real provider keys:

```bash
docker run --rm --env-file .env -p 8000:8000 multi-provider-llm-starter-kit
```

## GET /health

Checks whether the API is running.

### Request

```bash
curl http://127.0.0.1:8000/health
```

### Example Response

```json
{
  "status": "ok",
  "app_name": "Multi-Provider LLM Starter Kit",
  "environment": "development"
```

## POST /generate

Generates a standard non-streaming LLM response.

### Request with Mock Provider

```bash
curl -X POST "http://127.0.0.1:8000/generate" \
  -H "Content-Type: application/json" \
  -d '{"provider":"mock","prompt":"Explain provider abstraction in one paragraph."}'
```

### Example Response

```json
{
  "provider": "mock",
  "response": "This is a mock LLM response. In production, this would be generated by a real provider such as Gemini, Groq, or Mistral."
}
```

### Request with Gemini

```bash
curl -X POST "http://127.0.0.1:8000/generate" \
  -H "Content-Type: application/json" \
  -d '{"provider":"gemini","prompt":"Explain provider abstraction in one paragraph."}'
```

### Request with Groq

## POST /generate/stream

Streams generated text chunk by chunk.

This is useful for chatbots, copilots, RAG assistants, and long responses where the user should see output as it is generated.

### Request with Mock Provider

```bash
curl -N -X POST "http://127.0.0.1:8000/generate/stream" \
  -H "Content-Type: application/json" \
  -d '{"provider":"mock","prompt":"Explain streaming responses in one paragraph."}'
```

The `-N` flag tells curl not to buffer the response, making streaming easier to observe.

### Request with Gemini

```bash
curl -N -X POST "http://127.0.0.1:8000/generate/stream" \
  -H "Content-Type: application/json" \
  -d '{"provider":"gemini","prompt":"Explain streaming responses in one paragraph."}'
```

## POST /complaints/analyse

Analyses a banking complaint and returns structured JSON.

### Request

```bash
curl -X POST "http://127.0.0.1:8000/complaints/analyse" \
  -H "Content-Type: application/json" \
  -d '{"provider":"mock","complaint":"My QR payment was deducted but the merchant says they did not receive it."}'
```

### Example Response

```json
{
  "provider": "mock",
  "summary": "Customer reports a deducted QR payment that the merchant did not receive.",
  "category": "payment_dispute",
  "urgency": "high",
  "needs_human_review": true,
  "recommended_action": "Check transaction status and merchant settlement records."
}
```

## Why Structured Output Matters

Structured output can be used by:

- complaint management systems
- support dashboards
- escalation queues
- workflow automation tools
- reporting pipelines

Instead of returning only a paragraph, the model output becomes usable data.
